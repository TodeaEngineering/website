Built as an open-source Kubernetes-native policy engine, Kyverno expresses policy as YAML, and increasingly CEL, evaluated at admission and continuously audited against running workloads. Trusted in production by Coinbase, LinkedIn, and Spotify, and serving as the default policy engine for the U.S. Department of Defense's Platform One, Kyverno officially reached [CNCF Graduation](https://github.com/cncf/toc/issues/1967) at KubeCon Amsterdam in March 2026.

## What Kyverno actually is

Kyverno is an open-source Kubernetes-native policy engine that expresses policy as Kubernetes resources. Unlike OPA/Gatekeeper, which asks you to learn Rego before you can write a rule, Kyverno policies are YAML, and support CEL written against the same API objects you already understand.

The developer/sre team write a `ValidatingPolicy` the same way they write a `Deployment`, applies it with `kubectl`, and gets violations back as standard Kubernetes events and `PolicyReport` resources. There is no separate language, no separate query model, no separate mental stack.

## How it works

Kyverno is built on Kubernetes' dynamic admission controller architecture, with a background scanner layered on top. Policies evaluate at three points in a resource's lifecycle:

1. **Admission**: When a developer or GitOps tool, CI/CD pipeline submits a resource, the API server fires admission webhooks. Kyverno's webhook receives the request, evaluates matching policies, and returns `allow`, `deny`, or a mutated resource, in the request path.
2. **Reconciliation**: Background controllers watch for resources that should exist (cleanup policies, generation rules) and reconcile them on a loop, separate from admission.
3. **The new reality**: Kyverno's reporting subsystem scans existing resources against current policies on a schedule, surfacing violations in workloads that were created before a policy existed or that drifted afterward.

The application code is unchanged. Every resource that flows through the API server is checked, mutated, and recorded.

## Policy types

Kyverno introduced five CEL-native CRDs across two releases ŌĆö `ValidatingPolicy` and `ImageValidatingPolicy` in v1.14 (April 2025), then `MutatingPolicy`, `GeneratingPolicy`, and `DeletingPolicy` in v1.15 (July 2025) ŌĆö and promoted the whole group from `v1beta1` to `v1` together in 1.17. They are separate kinds, and each one of them tells the API server what to do with the result:

- **`ValidatingPolicy`**: Validates resources against a CEL expression. The CEL-native successor to the validate rule in `ClusterPolicy`, which still ships and keeps working.

```yaml
apiVersion: policies.kyverno.io/v1
kind: ValidatingPolicy
metadata:
  name: require-team-label
spec:
  validationActions: [Deny]
  matchConstraints:
    resourceRules:
      - apiGroups: [""]
        apiVersions: ["v1"]
        operations: ["CREATE", "UPDATE"]
        resources: ["pods"]
  validations:
    - expression: "has(object.metadata.labels) && 'team' in object.metadata.labels"
      message: "Pods must carry a 'team' label."
```

- **`MutatingPolicy`**: Mutates resources during admission using CEL-evaluated `applyConfiguration` (server-side apply) or `jsonPatch` expressions, mirroring upstream Kubernetes `MutatingAdmissionPolicy`. Strategic-merge and raw JSON 6902 patches remain available in the legacy `ClusterPolicy` mutate rule.

```yaml
apiVersion: policies.kyverno.io/v1
kind: MutatingPolicy
metadata:
  name: default-run-as-non-root
spec:
  matchConstraints:
    resourceRules:
      - apiGroups: [""]
        apiVersions: ["v1"]
        operations: ["CREATE"]
        resources: ["pods"]
  mutations:
    - patchType: ApplyConfiguration
      applyConfiguration:
        expression: >
          Object{
            spec: Object.spec{
              securityContext: Object.spec.securityContext{
                runAsNonRoot: true
              }
            }
          }
```

- **`GeneratingPolicy`**: Generates downstream resources when a trigger resource is created, namespace defaults, network policies, RBAC bindings.

```yaml
apiVersion: policies.kyverno.io/v1
kind: GeneratingPolicy
metadata:
  name: default-deny-on-namespace
spec:
  matchConstraints:
    resourceRules:
      - apiGroups: [""]
        apiVersions: ["v1"]
        operations: ["CREATE"]
        resources: ["namespaces"]
  generate:
    - expression: >
        [
          Resource{
            apiVersion: "networking.k8s.io/v1",
            kind: "NetworkPolicy",
            metadata: Object.metadata{
              name: "default-deny",
              namespace: object.metadata.name
            },
            spec: Object.spec{
              podSelector: Object.spec.podSelector{},
              policyTypes: ["Ingress", "Egress"]
            }
          }
        ]
```

- **`ImageValidatingPolicy`**: Validates container images against trusted attestors (Cosign, Notary, sigstore TUF) and in-toto attestations as first-class inputs. SBOMs are supported as one attestation payload type.

```yaml
apiVersion: policies.kyverno.io/v1
kind: ImageValidatingPolicy
metadata:
  name: require-signed-images
spec:
  validationActions: [Deny]
  matchConstraints:
    resourceRules:
      - apiGroups: [""]
        apiVersions: ["v1"]
        operations: ["CREATE", "UPDATE"]
        resources: ["pods"]
  matchImageReferences:
    - glob: "ghcr.io/my-org/*"
  attestors:
    - name: my-org-cosign
      cosign:
        key:
          data: |
            -----BEGIN PUBLIC KEY-----
            MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAE...
            -----END PUBLIC KEY-----
  validations:
    - expression: "images.containers.all(image, image.verified == true)"
      message: "All container images must be signed by the my-org Cosign key."
```

- **`DeletingPolicy`**: Deletes resources matching a selector on a schedule. Useful for ephemeral environments and cleanup workflows.

```yaml
apiVersion: policies.kyverno.io/v1
kind: DeletingPolicy
metadata:
  name: cleanup-preview-pods
spec:
  schedule: "0 2 * * *"
  matchConstraints:
    resourceRules:
      - apiGroups: [""]
        apiVersions: ["v1"]
        resources: ["pods"]
    namespaceSelector:
      matchExpressions:
        - key: kubernetes.io/metadata.name
          operator: In
          values: ["preview-feature-a", "preview-feature-b"]
```

Each of the five kinds also ships with a `NamespacedŌĆ”` equivalent (`NamespacedValidatingPolicy`, `NamespacedMutatingPolicy`, `NamespacedGeneratingPolicy`, `NamespacedImageValidatingPolicy`, `NamespacedDeletingPolicy`). This matters for multi-tenancy: tenant teams can author and manage policy inside their own namespace without cluster-admin, while platform owners keep the cluster-scoped kinds for guardrails that have to apply everywhere.

A close look at the CEL means these have roughly the maturity of the analogous Validating Admission Policy in upstream Kubernetes, with the API server doing more of the work. Kyverno wraps that with the surrounding policy engine, reporting, exceptions, generation, image validation, that VAP alone does not provide.

The legacy `ClusterPolicy` and `CleanupPolicy` kinds keep working today, but they are on a clock. [Kyverno 1.17](https://kyverno.io/blog/2026/02/02/announcing-kyverno-release-1.17/) officially marked them as deprecated; v1.18 and v1.19 ship critical fixes only; v1.20 (October 2026) is the planned removal. Treat the migration to the CEL-native CRDs as a dated workstream, not an "eventually", new policies should be written against the CEL-native kinds unless there is a specific reason not to.

## When to use ValidatingAdmissionPolicy instead

Kubernetes 1.30+ ships `ValidatingAdmissionPolicy` as a built-in, no-controller-required CEL validator. For some teams, that is enough. For most, it isn't.

| Capability / Feature | Built-in `ValidatingAdmissionPolicy` | Kyverno `ValidatingPolicy` (CEL) |
|---|---|---|
| Primary function | Synchronous validation at admission | Validation, mutation, generation, deletion |
| Enforcement points | Admission only | Admission, background scans, pipelines, CLI |
| Payloads | Kubernetes objects | Kubernetes objects, plus any JSON/YAML |
| CEL library | Basic | Extended (HTTP calls, image verification, Kubernetes lookups) |
| External data | Ō£Ģ | Kubernetes resources or HTTP calls |
| Policy bindings | Manual | Automatic |
| Background scans | Ō£Ģ | Periodic, plus on policy change |
| Reporting & audit | Ō£Ģ | `PolicyReport` CRDs / OpenReports API |
| Exceptions | Ō£Ģ | Fine-grained `PolicyException` |
| Image signature verification | Ō£Ģ | Ō£ō |
| Auto-generation | Ō£Ģ | Pod controllers, `ValidatingAdmissionPolicy` |
| Testing | Ō£Ģ | Kyverno CLI (unit), Chainsaw (e2e) |
| Architecture | Built into the API server (no controller) | Requires Kyverno controller |

If the only thing you need is a synchronous validate at admission time, VAP is the lighter-weight option and you should use it. If you need any of the rest, you need a policy engine, and Kyverno is the one with the smallest gap between "policy as YAML" and "policy as code".

## Operational reality

- **Rollout, not install.** Kyverno in `Audit` mode on day one is the right default. The temptation is to ship a policy in `Enforce` and discover at 2am that a controller you forgot about can't reconcile because its ServiceAccount fails your label policy. Run in Audit for at least a week, watch the `PolicyReport` resources, fix the violations you find in the workloads, then flip to Enforce. 

- **Mutation is not a security control.** `MutatingPolicy` is excellent for setting defaults, image pull secrets, resource requests, standard labels, and a poor substitute for validation. A user who can submit a pod spec can submit one your mutation never sees, because admission ordering and exemptions exist. Anything that has to be true gets a `ValidatingPolicy`. 

## A practical recommendation

If you are evaluating a policy engine for a Kubernetes platform and you do not already have a strong reason to pick OPA/Gatekeeper, pilot Kyverno first. The install fits in an afternoon. If it does what you need, you just saved a quarter of operational work. If it doesn't, you have a clear, documented reason to go somewhere else.

## Frequently Asked Questions

### What is Kyverno, and where does it sit in Kubernetes admission flow?

Kyverno is a Kubernetes-native policy engine that runs as an admission controller, intercepting every API request via webhooks to validate, mutate, or generate resources before they are persisted.

### Should I start with legacy ClusterPolicy or the newer CEL-based policy types?

Use the CEL-based types (ValidatingPolicy, MutatingPolicy, GeneratingPolicy, ImageValidatingPolicy) for new work. ClusterPolicy still works but is deprecated as of Kyverno 1.17.

### How do Audit and Enforce differ, and what do background scans do?

Enforce rejects failing requests at admission; Audit admits them but records the failure. Background scans re-evaluate existing resources on a schedule and write results to PolicyReports.

### How do generate rules work, and how do I keep generated resources synchronized?

Generate rules create downstream resources from a trigger event. With `synchronize: true`, manual edits revert and source changes propagate. With `false`, Kyverno creates once and stops.
