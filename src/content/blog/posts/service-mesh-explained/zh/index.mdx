如果你在 Kubernetes 上运行的微服务超过寥寥几个，你的开发者很可能已经在不知不觉中自己实现了服务网格的某些部分。HTTP 客户端中的重试逻辑，由脚本轮换的 TLS 证书，追踪服务间调用的自定义 Prometheus exporter，某处定义超时的 YAML 文件。

每个服务团队都在以略微不同的方式重新实现相同的模式。服务网格把这一切从应用代码中移走，交给平台团队拥有的基础设施。

## 不该由开发者承担的职责

把单体拆成微服务时，原本在进程内的所有东西 — 错误处理、重试、超时、认证、可观测性 — 也一并被拆散。没有服务网格的话，每个团队都要重新构建这些模式：

- **重试和超时逻辑**散落在每个 HTTP 客户端中，常常配置错误
- **TLS 证书管理**配上头部里到处传递的共享密钥
- **服务间认证**用 API token 临时凑合
- **分布式追踪埋点**手工添加到每一个端点
- **熔断器**在服务之间被复制，带着细微的差异
- **负载均衡和重试预算**散落在应用代码各处
- **网络策略**写成无人敢动的 IP 白名单

把这些乘以平台中的每一个服务、你交付的每一种语言、以及必须维护它的每一个团队。代价巨大，一致性糟糕，可靠性受损。

服务网格把这些负担彻底从开发者身上拿掉：

![服务网格为开发者卸下的职责](/blog/service-mesh-explained/zh-duties.svg)

开发者写业务逻辑。网格处理那些原本属于横切关注点的所有管道工作。

## 工作原理：代理架构

服务网格通过一个看似简单的模式实现这一切。两个组件：

1. **数据平面** — 与每个服务实例并行运行的轻量网络代理（**sidecar 模式**），拦截每一字节的入站和出站流量。
2. **控制平面** — 集中式组件，配置代理并聚合其遥测数据。

当服务 A 想调用服务 B 时，A 不会直接调用 B。它调用本地代理。代理处理 mTLS、重试、超时、追踪和路由，然后把请求转发给 B 的代理。B 的代理执行策略并转发给 B。

从应用角度看，什么都没变 — 它仍然是向 `localhost` 发起一个普通的 HTTP 调用。从运维角度看，每一字节流量现在都可观测、可保护、可控制。无需修改一行应用代码。

![服务网格的代理架构](/blog/service-mesh-explained/zh-service-mesh.svg)

这就是整个基础：把共享的基础设施关注点从应用代码中移出，放到由平台团队拥有的、按服务部署的代理中。

## 在平台层面你能得到什么

部署得当的服务网格在平台层面提供（无需改动应用）：

- **服务之间的双向 mTLS。** 网格处理证书的签发、轮换和吊销。应用团队的代码库中没有 PKI 代码。
- **基于密码学证书的工作负载身份**，而非 IP 地址或共享密钥。零信任网络的基础。
- **流量管理** — 重试、超时、熔断、金丝雀发布 — 通过配置声明。
- **分布式追踪**免费提供。每个请求自动传播一个 trace ID。
- **一致的指标** — 请求率、错误率、p50/p95/p99 延迟 — 适用于每个服务，无需任何插桩工作。
- **多集群服务发现**，A 集群的服务可以像调用邻居一样调用 B 集群的服务。

## 实际何时需要

并非每个 Kubernetes 平台都需要服务网格。需要它的信号：

- 互相通信的服务**超过 10 个**
- 出于合规需要强制执行**零信任网络**（金融、政府、医疗）
- 在**多个集群或区域**运行，需要跨集群服务发现
- 你已经厌倦了**没有分布式追踪的调试**
- 服务间**不一致的重试与超时行为**导致故障

如果你只有三个服务且都在一个集群中运行，你可能还不需要网格。解决你已有的问题，而不是可能会有的问题。

## 常见陷阱

服务网格的引入会以可预见的方式失败：

1. **当作开关来对待。** 安装网格是简单的部分。运维 — 管理证书颁发机构、调试代理问题、处理升级 — 才是难的部分。请规划运维投入。
2. **按功能而非匹配度选择。** 网格生态系统（Istio、Linkerd、Consul、Cilium service mesh）有非常不同的运维特征。适合 Google 规模的网格对大多数企业来说都过度。
3. **忘记延迟代价。** 每个请求现在要经过两个额外的代理。现代网格在 p50 上增加 1–3ms，但在长调用链上会累积。
4. **跳过可观测性工作。** 服务网格产生大量遥测数据。如果没有仪表盘和 SLO，这些数据毫无用处。

## 未来之路

服务网格正向更简单、更聚焦的工具演进。过去三年的趋势：

- **节点级代理**取代 pod 级 sidecar（Istio Ambient、Cilium） — 内存成本更低，但安全权衡不同
- **基于 eBPF 的网格化**用于 L4 流量，完全绕开用户态代理
- **与平台工程更紧密的集成** — Backstage 插件、IDP 暴露、GitOps 原生配置
- **零信任作为首要成果**，而非众多功能之一

技术正在趋于稳定。这种模式 — 将横切关注点移出应用代码 — 将长期存在。

## 实用建议

如果你正在大规模运行生产 Kubernetes，但尚未引入服务网格，请从两周的评估开始：

1. 梳理你的服务间流量，识别价值最高的问题（安全？可观测性？流量管理？）
2. 在非生产集群中试点一个网格四周
3. 基于该经验做运维决策，而不是基于基准测试

服务网格是云原生团队能做出的最具杠杆效应的平台投资之一 — 同时也是反射性引入时最痛苦的之一。决定因素是运维成熟度，而非功能清单。
