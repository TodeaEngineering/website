Kubernetes에서 마이크로서비스를 몇 개 이상 운영하고 있다면, 여러분의 개발자들은 이미 자기도 모르는 사이에 서비스 메시의 일부를 직접 구현하고 있을 가능성이 높습니다. HTTP 클라이언트의 재시도 로직, 스크립트로 갱신되는 TLS 인증서, 서비스 간 호출을 추적하는 Prometheus 익스포터, 어딘가에 정의된 타임아웃 YAML 파일.

각 서비스 팀은 같은 패턴을 조금씩 다른 방식으로 다시 구현하게 됩니다. 서비스 메시는 이 모든 것을 애플리케이션 코드에서 빼내어, 플랫폼 팀이 소유하는 인프라로 옮깁니다.

## 개발자가 떠안지 말아야 할 일들

모놀리스를 마이크로서비스로 분산시키면, 프로세스 내부에 있던 모든 것 — 오류 처리, 재시도, 타임아웃, 인증, 옵저버빌리티 — 도 함께 분산됩니다. 메시가 없다면, 모든 팀이 같은 패턴을 다시 만듭니다:

- **재시도 및 타임아웃 로직**이 모든 HTTP 클라이언트에, 종종 잘못 구성된 채로
- **TLS 인증서 관리**가 헤더로 전달되는 공유 시크릿과 함께
- **서비스 간 인증**이 API 토큰으로 임시방편처럼 처리됨
- **분산 추적 계측**이 모든 엔드포인트에 수동으로 추가됨
- **서킷 브레이커**가 미묘한 차이를 가진 채 서비스 간에 복사됨
- **로드 밸런싱 및 재시도 예산**이 애플리케이션 코드 곳곳에 흩어짐
- **네트워크 정책**이 아무도 손대지 않는 IP 허용 목록으로 표현됨

이를 플랫폼의 모든 서비스, 모든 언어, 그리고 이를 유지보수해야 하는 모든 팀으로 곱해 보세요. 비용은 막대하고, 일관성은 떨어지며, 신뢰성은 손상됩니다.

서비스 메시는 이러한 부담을 개발자에게서 완전히 제거합니다:

![서비스 메시가 개발자의 부담을 덜어주는 영역](/blog/service-mesh-explained/ko-duties.svg)

개발자는 비즈니스 로직을 작성합니다. 메시는 횡단적 배관 작업이었던 모든 것을 처리합니다.

## 작동 방식: 프록시 아키텍처

서비스 메시는 겉보기에는 단순한 패턴으로 이를 구현합니다. 두 가지 컴포넌트:

1. **데이터 플레인** — 모든 서비스 인스턴스 옆에서 실행되는 경량 네트워크 프록시(**사이드카 패턴**)로, 들어오고 나가는 모든 트래픽 바이트를 가로챕니다.
2. **컨트롤 플레인** — 프록시를 구성하고 텔레메트리를 집계하는 중앙 컴포넌트.

서비스 A가 서비스 B를 호출하려고 할 때, A는 B를 직접 호출하지 않습니다. 자체 로컬 프록시를 호출합니다. 프록시가 mTLS, 재시도, 타임아웃, 추적, 라우팅을 처리한 후, B의 프록시로 요청을 전달합니다. B의 프록시는 정책을 적용하고 B로 전달합니다.

애플리케이션 입장에서는 변한 것이 없습니다 — 여전히 `localhost`로 일반 HTTP 호출을 합니다. 운영자 입장에서는 모든 트래픽 바이트가 이제 관측 가능하고, 보안되며, 제어할 수 있습니다. 애플리케이션 코드 한 줄도 변경하지 않고요.

![서비스 메시 프록시 아키텍처](/blog/service-mesh-explained/ko-service-mesh.svg)

이것이 전체의 기반입니다: 공유 인프라 관심사를 애플리케이션 코드에서 빼내어, 플랫폼 팀이 소유하는 서비스별 프록시로 옮기는 것.

## 플랫폼 수준에서 무엇을 얻는가

올바르게 배포된 서비스 메시는 애플리케이션 변경 없이 다음을 제공합니다:

- **모든 서비스 간 mTLS.** 메시가 인증서 발급, 갱신, 폐기를 처리합니다. 앱 팀의 저장소에 PKI 코드가 없습니다.
- **워크로드 아이덴티티**는 IP 주소나 공유 시크릿이 아닌 암호화 인증서를 기반으로 합니다. 제로 트러스트 네트워킹의 기반입니다.
- **트래픽 관리** — 재시도, 타임아웃, 서킷 브레이킹, 카나리 릴리스 — 가 구성으로 선언됩니다.
- **분산 추적**이 무료로 제공됩니다. 모든 요청에 추적 ID가 자동으로 전파됩니다.
- **일관된 메트릭** — 요청 비율, 오류 비율, p50/p95/p99 지연 시간 — 가 계측 작업 없이 모든 서비스에 대해 제공됩니다.
- **멀티 클러스터 서비스 디스커버리**, 클러스터 A의 서비스가 클러스터 B의 서비스를 마치 이웃처럼 호출할 수 있습니다.

## 실제로 언제 필요한가

모든 Kubernetes 플랫폼이 서비스 메시를 필요로 하는 것은 아닙니다. 필요하다는 신호:

- 서로 통신하는 서비스가 **약 10개 이상**입니다
- 컴플라이언스를 위해 **제로 트러스트 네트워킹**을 강제해야 합니다 (금융, 정부, 의료)
- **여러 클러스터나 리전**에 걸쳐 운영하며 클러스터 간 서비스 디스커버리가 필요합니다
- **분산 추적 없이 디버깅하는 데** 지쳤습니다
- 서비스 전반에 걸친 **일관성 없는 재시도 및 타임아웃 동작**이 장애를 일으킵니다

서비스가 3개이고 모두 한 클러스터에서 실행된다면, 아직 메시가 필요하지 않을 가능성이 높습니다. 있을지도 모르는 문제가 아니라, 실제로 가지고 있는 문제를 해결하세요.

## 흔한 함정

서비스 메시 도입은 예측 가능한 방식으로 실패합니다:

1. **스위치처럼 다루기.** 메시 설치는 쉬운 부분입니다. 운영 — 인증 기관 관리, 프록시 문제 디버깅, 업그레이드 처리 — 이 어려운 부분입니다. 운영 투자를 계획하세요.
2. **적합성이 아닌 기능으로 선택하기.** 메시 생태계(Istio, Linkerd, Consul, Cilium 서비스 메시)는 매우 다른 운영 프로필을 가지고 있습니다. Google 규모에서 동작하는 메시는 대부분의 기업에는 과합니다.
3. **지연 시간 비용을 잊기.** 모든 요청이 이제 두 개의 추가 프록시를 거칩니다. 현대 메시는 p50에 1~3ms를 추가하지만, 긴 호출 체인 전체에서 누적됩니다.
4. **옵저버빌리티 작업 건너뛰기.** 서비스 메시는 엄청난 텔레메트리를 생성합니다. 대시보드와 SLO가 없으면 데이터는 무용지물입니다.

## 앞으로의 길

서비스 메시는 더 단순하고 집중된 도구로 향하고 있습니다. 지난 3년의 동향:

- **노드별 프록시**가 파드별 사이드카를 대체 (Istio Ambient, Cilium) — 메모리 비용은 낮지만, 보안 트레이드오프가 다릅니다
- **eBPF 기반 메싱**으로 L4 트래픽 처리, 사용자 공간 프록시를 완전히 우회
- **플랫폼 엔지니어링과의 긴밀한 통합** — Backstage 플러그인, IDP 노출, GitOps 네이티브 구성
- **제로 트러스트가 부수적 기능이 아닌 핵심 결과**로 자리잡음

기술은 안정적인 모습으로 정착하고 있습니다. 패턴 — 횡단 관심사를 애플리케이션 코드 밖으로 옮기는 것 — 은 앞으로도 계속될 것입니다.

## 실용적인 권장사항

대규모로 프로덕션 Kubernetes를 운영하고 있는데 아직 서비스 메시가 없다면, 2주 평가로 시작하세요:

1. 서비스 간 트래픽을 매핑하고 가장 가치 있는 문제를 식별합니다 (보안? 옵저버빌리티? 트래픽 관리?)
2. 비프로덕션 클러스터에서 4주 동안 하나의 메시를 파일럿합니다
3. 벤치마크가 아닌 그 경험을 바탕으로 운영 결정을 내립니다

서비스 메시는 클라우드 네이티브 팀이 할 수 있는 가장 레버리지가 큰 플랫폼 투자 중 하나입니다 — 그리고 반사적으로 도입할 때 가장 고통스러운 것 중 하나이기도 합니다. 결정 요인은 기능 체크리스트가 아닌 운영 성숙도입니다.
